# Demo 设计思路与说明：本地生活执行 Agent

## 1. 设计目标

本 Demo 面向“周末本地生活安排”命题，核心不是简单推荐地点，而是把用户一句自然语言需求转成可确认、可调整、可模拟执行的完整任务。系统需要同时覆盖不同人群、不同城市、不同约束和异常情况，并保证结果符合现实生活中的时间、距离、预算和体力规律。

项目当前定位为产品原型：地点候选可使用本地 POI 种子、可选 LLM 补全或 Mock 检索结果；天气、路线耗时、余位、订座、下单、通知和提醒等实时字段均按 Mock API 模拟，不声称已经接入真实生活服务平台。

## 2. 完整性设计

系统从四类典型用户出发设计规划策略：

- 亲子家庭：强调儿童友好、距离短、体力负担低、餐饮健康和节奏稳定。
- 朋友小聚：强调多人适配、聊天空间、餐厅容量、晚餐衔接和群消息通知。
- 情侣或拍照休闲：强调氛围、路线顺畅、停留时间和餐饮体验。
- 长辈同行：强调少走路、低噪音、可休息、交通方便和风险保守。

地域覆盖采用分层策略。热门城市优先使用真实 POI 种子，冷门城市或无法确认地点时退回类型化候选，例如“室内亲子乐园”“近距离商场餐厅”“低强度公园散步”。这样可以保证现场随机输入时流程不中断，同时避免把无法验证的地点包装成真实 API 结果。

每个输出方案都必须包含：时间线、活动、餐饮、交通、预算、推荐理由、风险提示、备选节点和执行动作。规划时先检查硬约束，例如时间窗、距离、人数、座位、票量和高影响动作确认；再用软约束打分，例如少排队、低预算、健康饮食、室内外偏好和人群适配。时间分配会保留出发、转场、等待和用餐缓冲，避免出现“1 小时玩很远地点再吃饭”这类不符合自然规律的安排。

## 3. 端到端调用与稳定性

主链路采用稳定优先的设计：

```text
用户输入 -> 意图理解 -> Mock 工具查证 -> 多候选合成 -> 可行性校验 -> 用户确认 -> 模拟执行队列
```

后端、LLM 和 LangGraph 只作为增强链路：可用时增强意图识别、反馈复盘和候选记忆；不可用、超时或置信度不足时，前端立即回退到本地规则与 Mock Tools。这样可以保证端到端 Demo 不依赖真实网络成功率，也不会因为外部接口波动导致现场链路失败。

Mock API 不是随意兜底，而是有明确语义：它负责模拟天气、路线、活动、餐厅、票务、预约和通知等工具结果，并支持可控的失败、超时、满座、无票和重试场景。所有高影响动作，例如订座、购票、下单和发消息，都必须在用户确认后才进入模拟执行队列。

## 4. 异常与兜底策略

系统内置异常处理，而不是只在理想输入下工作：

- 下雨：户外活动降级为室内活动，并重新计算路线。
- 餐厅满座：替换同商圈、同预算、同人群适配的餐厅。
- 活动无票：替换同时间窗、同半径、同强度活动。
- 人数变化：重新检查票量、座位、预算和交通。
- 孩子累了或长辈不适：降低活动强度，缩短移动距离。
- 信息不足：使用可解释假设生成可调整方案；若缺失信息影响安全或关键可执行性，则软追问。
- 外部增强失败：保留最后稳定方案，切换到本地规则，不渲染不可信数据。

兜底目标不是“永远给答案”，而是在风险可控时继续推进，在风险不可控时明确提示用户确认或调整。

## 5. 创新性与交互设计

交互上不把体验限制为 Chatbot。用户可以先用一句话启动规划，再通过卡片、时间线、候选切换、假设编辑和执行中心完成后续决策。

核心交互包括：

- 候选方案卡片：展示 2 到 3 个服务包，突出推荐理由、风险和差异。
- 时间线视图：把出发、活动、转场、用餐和提醒排成可检查的日程。
- 假设栏：暴露系统默认的人数、预算、区域和时间，允许用户直接修改。
- 局部重排：只替换活动、餐厅或交通，不必每次全量重新生成。
- 执行中心：把订座、排队、买票、通知和提醒拆成可追踪动作。
- 本地协同反馈：可把方案发给家人朋友查看和反馈，发起人确认后再执行。

这种设计让用户看到 Agent 的判断过程，而不是只看到一段推荐文案。它也更适合比赛展示：评委可以直观看到理解、查证、校验、重排和执行闭环。

## 6. Demo 展示重点

1. 输入亲子家庭场景，展示系统如何理解孩子、距离、减脂餐饮和 4 小时时间窗。
2. 展示工具查证和多候选方案，说明方案来自天气、活动、餐厅、路线和可预约状态组合。
3. 切换一个异常，例如餐厅满座或下雨，展示局部重排。
4. 用户确认后进入模拟执行队列，展示订座、通知和提醒的端到端闭环。
5. 再输入模糊或冲突场景，展示系统不会盲目承诺，而是通过假设、追问或降级保持可靠。

最终希望凸显的能力是：系统能完整解决周末本地生活安排问题，同时对数据来源、外部调用、异常恢复和高影响动作边界保持清楚、稳健、可解释。
