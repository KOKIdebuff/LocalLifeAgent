# Lessons

## 2026-06-06 候选费用必须相对当前已采用状态计算

### issue

交通候选首次采用打车后再切换公交，如果把公交费用继续当作相对原方案的新增费用，
会造成预算重复累加；恢复原交通也无法正确扣回已采用费用。

### root_cause

预览差异只比较候选和原始状态，没有把“当前已采用候选”作为差异基线。

### fix

交通预算差异改为 `目标候选费用 - 当前已采用候选费用`。时间差继续根据当前行程段
文本计算，恢复原方案时相应扣回已采用交通费用。

### prevention

所有可连续采用的候选切换器必须区分 original、preview 和 adopted 三个基线；
差异计算始终相对当前 committed 状态，不能固定相对首次生成状态。

## 2026-06-06 兼容入口的删除必须有明确的人验闸门

### issue

仅写“兼容保留”仍可能被后续实现者理解为新功能上线后即可自动删除，无法表达用户要求
亲自体验审查的产品验收条件。

### root_cause

兼容策略只有技术状态，没有删除责任人和明确的删除前置条件。

### fix

为 `refresh_block` 增加机器可读 `deletionGate`：只有用户亲自体验新流程并明确批准后
才能删除。同时将交通从时间线附属文案升级为绑定行程段的独立可操作卡。

### prevention

涉及旧入口删除、数据迁移或兼容层下线时，契约必须同时声明保留状态、行为限制、
验收责任人和显式删除闸门，不能只依赖“后续清理”文字。

## 2026-06-06 预览导航和方案提交必须分离

### issue

旧“换一换”契约把点击动作、候选生成、局部替换和 Main 更新合并在一起，导致用户
无法稳定返回上一个候选，也无法明确判断当前内容是预览还是已采用。

### root_cause

状态模型只保存了重排前快照，没有保存候选列表、当前位置、原始候选、已采用候选和
受影响时间线；交互动作也只有笼统的 `refresh_block`。

### fix

将“换一换”升级为当前编辑会话内的线性候选项切换器。预览动作不修改 Main 和已保存
快照；采用、恢复原方案和采用后一次撤销使用明确动作，并在提交前校验时间、预算、
风险和 schema。

### prevention

所有“浏览多个结果后选择一个”的功能都必须把 navigation state、preview state 和
commit state 分开建模。兼容入口可以保留，但不得继续隐式执行提交。

## 2026-06-05 目标能力不能冒充运行时有效能力

### issue

原 `x-runtimeP0Capabilities` 表达的是 V4 P0 目标，但 `supported` 的语义没有
明确写成“契约已冻结”，容易被误读为已经具备持久化、事件流和恢复点。

### root_cause

同一 capability 列表混合了产品路线图和运行时事实，没有为消费者提供明确的
权威可用性来源。

### fix

拆分为 `targetCapabilities` 和 `effectiveCapabilities`。目标侧
`status=supported` 明确表示契约已冻结；有效侧改用
`availability=available / degraded / unavailable`，并成为 UI 启用、降级和
fallback 的唯一依据。

### prevention

所有能力协商契约必须区分 desired、declared 和 effective。契约冻结状态和运行时
可用性不得复用同一个字段或枚举。

## 2026-06-05 互斥规则必须按字段存在性表达

### issue

旧 schema 只在 `feedback` 和 `memoryDecision` 都是对象时拒绝，两个字段同时
出现但其中一个为 `null` 时仍可能通过。

### root_cause

互斥条件混入了字段类型判断，没有直接表达“两个字段不得同时存在”。

### fix

使用 `not + required: [feedback, memoryDecision]` 按字段存在性拒绝，并增加
机器可读规则、非法 null 组合示例和契约测试。

### prevention

互斥、至少一个、恰好一个等结构约束应优先使用 JSON Schema 的字段存在性关键字；
字段类型校验留在各字段自己的 schema 中，避免二者耦合产生旁路。

## 2026-06-05 Adapter 方法名不能混淆事件意图与持久事件

### issue

RuntimeAdapter 需要公开 `submit_event`，但既有契约把“客户端不能直接写正式
Event”表达成了 `submit_command`。这导致方法名、Event Contract 和状态权威边界
互相冲突。

### root_cause

契约没有区分客户端提交的事件意图与服务端校验后生成的权威 Event。

### fix

保留 `submit_event` 作为 UI 无关的公共写入口，同时明确其输入不包含可信
`fromState`。服务端从持久 Session 读取状态，校验 lifecycle、transition、
`expectedVersion` 和 `idempotencyKey` 后，才生成并持久化正式 Event。

### prevention

后续契约必须分别命名 intent、command 和 persisted event；任何公共方法变更都要
同步检查 Session 模型、Event 模型、Capability、事实源和契约测试，不能只改方法枚举。

## 2026-05-05 V2 第一轮比赛增强

### issue

V1 方案评分容易多个候选同时接近满分，朋友局的餐厅无座恢复路径也不一定出现在默认推荐中，执行结果里跳过动作存在被 UI 表达成“完成”的风险。

### root_cause

原评分主要依赖单一总分，缺少维度化扣分；异常恢复只放在风险文案里，展示优先级不够高；执行队列状态只有成功、跳过和待确认的基础区分，最终摘要没有按状态分别展示。

### fix

新增 `scoreDetails`、`recommendationReasons` 和 `issueNotices`，把距离、同行适配、预约稳定性、预算、风险扣分和异常恢复显式展示；执行队列和最终摘要按成功、待确认、需人工确认、已跳过分别渲染。

### prevention

回归测试覆盖评分差异、朋友局无座替换、雨天降低户外方案、预算/距离约束、人工确认动作不得伪造成成功。后续改评分或队列展示时必须先运行 `npm.cmd test`。

## 2026-05-05 V3 本地生活执行 Agent

### issue

V2 虽然增强了 Agent 展示，但产品叙事仍偏“活动规划器”，本地生活服务平台语境不够强，缺少排队、订座、团购、优惠、加购和动态重排的完整服务包表达。

### root_cause

上一轮改动主要围绕评分解释和方案卡展示，底层数据仍是“活动 + 餐厅”的组合，没有把生活服务平台的交易闭环和闲时转化价值建模出来。

### fix

新增 `servicePackage`，把时间轴、订座、排队、买票、买团购、下单加购、通知、闲时指标和重排事件统一到方案内。页面改成执行驾驶舱，工具调用改为可读办事流。

### prevention

后续展示增强必须先判断是否服务于“帮用户把事情做完”。不要只堆推荐理由或评分维度；每个主线能力都应落到可确认动作、异常兜底或业务指标上。

## 2026-05-07 Agent Loop Trace 轻量编排

### issue

现有 V3 已有工具调用、服务包和重排能力，但比赛表达仍容易被理解成“顺序调用几个 Mock API”，不够直观体现 Planner、研究通道、合成、校验和重排的完整 Agent loop。

### root_cause

底层实现为了稳定采用单 Orchestrator + Mock Tools，没有把每一步的职责以结构化 trace 暴露给 UI。若直接引入重型多 Agent 框架，会增加现场不稳定性和解释成本。

### fix

新增 `agentLoopTrace`，用 planner、researchers、merger、verifier、revise 五阶段表达轻量可解释编排；UI 先展示 trace 摘要，再保留原始 Mock Tool 调用。视觉上参考 `awesome-design-md` 的 VoltAgent/Linear 方向，只在 trace 区域增强技术驾驶舱感。

### prevention

后续如果继续增强 Agent loop，优先补 trace、测试和演示解释，不要轻易改成真实并发 researcher 或引入新框架。必须持续说明 researchers 是逻辑研究通道，外部能力仍为本地 Mock。

## 2026-05-08 比赛冲刺展示增强

### issue

V3 已具备服务包和重排能力，但比赛现场仍需要更强的第一眼识别：硬指标、逻辑 DAG、活动无票、人数变化和分享闭环需要在 UI、Trace、执行队列与文档中同时可见。

### root_cause

上一版主要覆盖下雨、餐厅满座、孩子累了和预算太高，异常类型仍偏生活状态；Researchers 只用 finding 表达，缺少并行研究通道和 Mock 时延；发送消息只是文本草稿，未形成可展示的协同分享卡片。

### fix

新增比赛演示路径、Researchers `lanes`、活动无票重排、人数变化重排、重排校验执行动作和 Mock 分享卡片；首屏保留用户视角，不直接展示评审硬指标和 Mock 边界。同步更新 `DESIGN.md`、`DEMO_SCRIPT.md`、`README.md`、`progress.md`，并补充回归测试覆盖新接口和新异常。

### prevention

后续展示增强必须同时满足三处可见：方案卡可见、Trace 可见、执行队列可见。不要只在文档中描述异常能力；每个新增异常都应有测试证明不会伪造成成功。

## 2026-05-09 轻量 Agent Runtime 方案沉淀

### issue

当前意图识别主要依赖关键词和正则，稳定可控但泛化能力有限。用户希望比赛 Demo 更像真实智能 Agent，并具备可沉淀经验的“自进化”能力：自动总结错误、下次参考、减少重复误判。

### root_cause

V3 为了保证现场稳定性，采用单 Orchestrator + 本地 Mock Tools，没有接入真实大模型 API，也没有后端状态、记忆存储和复盘链路。若直接套用非官方泄露源码或重型多 Agent 框架，会带来合规、稳定性、成本和维护风险。

### fix

确定下一阶段采用自有轻量 Agent Runtime：只参考 Claude Code、Codex、Cursor 等公开产品形态中的通用思想，不使用任何非官方泄露源码。新增方向包括 LLM 结构化意图识别、schema 校验、缺信息追问、工具注册、状态机、复盘器和本地经验存储。当前 V3 代码不改，先在 `DESIGN.md` 和 `progress.md` 中记录 V4 方案边界。

### prevention

后续实现“自进化”时必须保持可控：经验沉淀只能作为下次规划上下文和回归测试来源，不允许自动改代码或自动执行高影响动作；对外表述应为“降低重复错误概率”，不能承诺“永不犯同样错误”。任何真实 LLM/API 接入都要保留本地规则校验、失败兜底和用户确认。

## 2026-05-10 V4 结构化记忆闭环

### issue

直接把用户反馈写成长期 lesson 风险较高，容易把一次性约束、弱证据推断或敏感信息污染长期记忆，也不利于用户审查和回滚。

### root_cause

早期 V4 方案只强调结构化复盘记忆，但没有把事实库、交换格式、审计日志和后续语义检索的职责分开，也没有候选确认、敏感分级和长期记忆准入流程。

### fix

将记忆体系调整为结构化闭环：事实库存储结构化记录，JSON 做 API 交换格式，JSONL 做审计日志，Vector DB 留作后续增强。反馈先生成候选记忆，用户采用/忽略/更正后才长期生效。敏感分级采用 L0-L3，默认只保存 L0/L1 的抽象偏好。

### prevention

后续不要把聊天记录等同于记忆，也不要无审查地自动记住一切。当前需求优先于历史记忆；L2/L3 高敏感信息默认不进入长期记忆；规划策略类经验应可追踪、可审查、可回滚。

## 2026-05-18 Mock/API/LLM 边界收口

### issue

“全 Mock”和“任意城市给出真实地点名”之间存在天然张力。如果严格全 Mock，地点名和随机城市覆盖会显得不真实；如果直接接真实地图或生活服务 API，又会引入 Key、限流、网络、超时和现场稳定性风险。同时，“真实地点名”容易被误解为营业时间、距离、排队、余位等实时类字段也都真实。

### root_cause

需求里同时存在比赛现场效果、全链路稳定、真实 LLM、随机城市输入和 Mock API 边界。此前文档只强调“外部能力 Mock”，没有把地点名、实时字段和执行动作分层定义，导致后续实现和讲解容易摇摆。

### fix

确定分层边界：执行动作全 Mock API；地点候选采用本地真实 POI 种子库 + LLM 补全 + 可行性校验；营业时间、距离、排队、余位、预约结果和路况统一由 Mock API 返回。技术路线收敛为 LangGraph 轻量编排 + 业务逻辑自研，推荐目标栈为 LangGraph + FastAPI + 本地 Mock 数据库 + 真实 LLM。LangGraph 只负责流程调度、状态流转、分支、重试、兜底和 trace，复杂业务规则仍放在清晰 Tool 和自研逻辑中。

### prevention

后续 UI、文档和讲解必须区分地点名候选、Mock API 实时类字段和 Mock API 执行动作。不能把 LLM 生成的地点名说成真实 API 查得，也不能把 Mock API 返回的余位、排队、距离、营业状态说成实时平台数据。实现上建议给工具结果增加来源标记，例如 `seed_poi`、`llm_poi_candidate`、`mock_search_result`、`mock_realtime_field`、`mock_execution`，并保留本地规则兜底和用户确认。

## 2026-05-18 产品化范围选择

### issue

最小比赛稳态更容易在 1 个月内交付，但用户明确选择偏产品化方案：30 个以上热门城市、每城每类 5 个以上真实 POI、带置信度/来源/替代推荐的 fallback、完整 Mock API 失败配置，以及接近产品级的 Web UI。

### root_cause

项目目标从“稳定演示比赛主链路”进一步提升为“更像真实产品的本地生活 Agent”。这会增强现场观感和评审追问承受力，但也会显著增加数据整理、Mock API 配置、前端交互和回归验证成本。

### fix

将偏产品化方案写入 `README.md`、`COMPETITION_BRIEF.md`、`DESIGN.md` 和 `progress.md`：热门城市覆盖 30+；每城每类 5+ POI；fallback 展示置信度、来源和替代推荐；Mock API 支持失败率、超时、重试和降级策略；UI 向筛选、地图感、详情页和多轮修改推进。

### prevention

执行时必须保持主链路优先。先保证 LangGraph 编排、POI 检索、Mock API、候选方案、异常重排和确认执行端到端稳定，再扩大城市和 UI 细节。Mock API 失败和超时必须可配置、可复现，不做不可控随机失败；产品级 UI 只服务核心闭环，不做无关装饰。

## 2026-05-24 Windows 测试临时目录隔离

### issue

`test_feedback_creates_lesson_and_retrieves_relevant_memory` 在本机运行时超时，表面看像 SQLite 或记忆检索卡住。

### root_cause

测试 helper 将临时目录硬编码到 `C:/tmp`。当前 Windows/沙箱环境对该路径的新建目录行为不稳定：PowerShell 会报权限拒绝，Python `tempfile.TemporaryDirectory(dir="C:/tmp")` 则可能挂住，导致测试还没进入 SQLite 建表就超时。

### fix

将测试临时目录改为仓库内 `.pytest_tmp`，并加入 `.gitignore`。SQLite 文件仍由每个测试的 `TemporaryDirectory` 隔离，测试完成后自动清理。

### prevention

单测不要依赖机器根目录、用户全局临时目录或外部固定路径。需要写文件或 SQLite 时，优先使用仓库内已知可写的测试专用目录，并确保该目录被 Git 忽略。

## 2026-05-25 验证基线必须绑定项目虚拟环境

### issue

旧进度曾把全局环境未能运行 `pytest` 写成后端功能验证未完成，但当前项目 `.venv` 中的 pytest 已可稳定执行并通过测试。

### root_cause

验证记录没有固定 Python 解释器与测试启动命令，导致“机器全局环境缺依赖”和“项目验证能力缺失”被混为一谈。

### fix

将验证基线统一为 `.\.venv\Scripts\python.exe -m pytest ...`，并在 `README.md`、`progress.md` 与 Spec Kit 验证文档中记录同一组可复跑命令和通过结果；PowerShell 下前端回归固定使用 `npm.cmd test`。

### prevention

后续更新进度时，测试结论必须附带实际使用的项目级命令、测试数量和告警状态；只有项目虚拟环境中的基线命令失败，才能将对应验证记为未完成或阻塞。

## 2026-05-27 候选更正与运行时失败状态必须同边界校验

### issue

候选记忆初次生成时会阻止 L2/L3 内容，但“更正后采用”曾可绕过同一隐私规则；同时，Runtime 会将无效决策记录成已忽略完成，SQLite 故障则直接返回服务错误。

### root_cause

更正路径复用了原候选的敏感等级而未重新构造候选；Runtime 的决策输入缺少嵌套校验，并将所有非采用结果合并到成功忽略分支；存储异常也没有按用户当前操作阶段建立恢复语义。

### fix

对候选更正重新执行敏感分类与候选结构生成，拒绝 L2/L3 和空白更正并保留 `pending`；将 Runtime 输入结构化，区分校验失败、业务拒绝和存储可恢复失败；在反馈与候选确认阶段发生存储故障时保留当前步骤供重试。长期记忆继续只承载偏好和复盘经验，未来第三方授权资料另设通道。

### prevention

任何能够写入长期记忆的新入口都必须复用统一隐私判定并覆盖绕过测试；任何新增 Runtime 失败状态都必须同时更新 schema、状态转移表、接口测试与 UI 消费约定，不能依靠前端猜测错误含义。

## 2026-05-28 长期记忆准入必须校验最终落库结构

### issue

独立审查发现，候选 `correct` 和异常 `adopt` 仍可通过裸手机号、授权码、token 或藏在 `key/scope` 里的敏感内容绕过长期记忆边界；敏感拒绝响应也可能回显旧候选原文；audit 写失败会在 SQLite 已提交后向客户端表现为失败。

### root_cause

隐私判断分散在候选生成、更正和采用路径中，且只覆盖部分文本字段；错误响应复用了完整候选对象；audit JSONL 被当作主流程的一部分执行，但它不具备和 SQLite 主事务一致的提交语义。

### fix

新增统一长期记忆准入层，对最终写入和索引的完整记忆结构做敏感/授权资料校验；候选决策错误响应只返回安全元数据；audit 改为 best-effort 非阻断日志，写失败只记录内部 warning，不改变已提交业务操作的成功响应。

### prevention

后续任何长期记忆写入入口都必须先构造最终落库结构，再调用统一准入层；schema 中的拒绝响应不能允许完整候选正文；如果未来需要正式审计，必须升级到同事务审计表或 outbox，不能依赖提交后追加文件来表达强一致审计。
## 2026-06-06 保存方案切片必须隔离首页状态与持久化快照

### issue

首页已有候选切换和 V5 adapter 状态，如果直接把保存、详情和恢复逻辑继续堆进 `app.js`，容易误保存全部候选、覆盖现有首页状态，并让版本与执行锁定边界无法独立测试。

### root_cause

首页状态服务于当前规划会话，而已保存方案需要稳定、可回放、可裁剪的生命周期数据。两者的数据保留策略不同：选中方案要完整，其他候选只能保留轻量摘要。

### fix

新增独立 `saved-plans.js` 作为前端生命周期边界，集中处理快照裁剪、候选摘要、版本、撤销、锁定、恢复策略和本地存储；`app.js` 只负责页面协调和复用现有视觉组件。

### prevention

后续协作者、执行中心和 Plan Branch 切片继续沿用领域边界，不把持久化对象直接等同于首页运行时状态。任何快照新增字段都先对照 `ui-contract.schema.json`，并补充“禁止保存字段”回归测试。
