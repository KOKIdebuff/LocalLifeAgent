# V2 最小改造清单

> 历史文档说明：本文记录的是项目从 V1 到 V2 的最小升级计划，用于保留当时的改造原则、弱项判断和验收标准。当前项目已经推进到 V3 美团闲时消费执行 Agent，当前状态以 `progress.md` 和 `COMPETITION_BRIEF.md` 的 V3 说明为准。

## 1. 改造原则

V2 的目标不是重写项目，而是在现有 V1 闭环上增强比赛展示效果。

优先级判断：

1. 先保证现有功能和测试不退化。
2. 优先做低风险、可展示、可回滚的增强。
3. 暂不引入真实外部 API、真实商家、真实地图、真实餐厅库存、真实支付/预约系统、登录或新前端框架。
4. 不爬取实时数据，所有外部信息用可控 Mock 数据表达。
5. 每次改动都应能用 `npm test` 或本地 Demo 验证。

## 2. 当前 V1 已满足能力

- 自然语言输入。
- 识别时间、同行关系、人数、偏好和限制。
- 信息不足时追问。
- 多场景支持：朋友、亲子、长辈、情侣、独自、同事。
- Mock Tool 调用：天气、活动、餐厅、路线、可预约状态。
- 生成多个候选方案并推荐。
- 餐厅无座时尝试替换备选。
- 约束冲突时生成近距离替代方案。
- 高影响动作需要确认。
- 执行队列可模拟预约、下单、发消息和提醒。
- `tests.js` 已覆盖主要核心回归。

## 3. 当前弱项

- 首屏价值主张不够强，评委不一定立刻看出它和普通推荐的区别。
- 评分只有总分，缺少维度化解释，多个方案可能同分。
- 工具调用链路虽然存在，但默认展示不够像“Agent 执行过程”。
- 异常处理已有雏形，但比赛现场不够突出。
- Mock 数据规模较小，部分结果容易显得固定。
- 执行阶段主要是成功路径，缺少部分失败、跳过、人工接管的展示。
- `agent-core.js` 逻辑集中，后续大扩展会增加维护风险。

## 4. P0：先补稳定性和可信度

### P0-1 补充回归测试

目标：保护现有逻辑，避免 V2 展示增强引入回归。

建议新增测试：

- 同事局。
- 独自探索。
- 天气不适合户外时降低户外方案。
- 预算或距离约束更强的输入。
- 执行动作缺少可预约时段时必须跳过，不得伪造成功。

影响范围：

- `tests.js`
- 少量 `agent-core.js` 逻辑修正，如果测试暴露问题。

验收：

```powershell
npm test
```

### P0-2 修正评分同质化

目标：让方案分数更可信，避免多个方案都接近满分。

建议做法：

- 保留总分。
- 增加内部维度分或解释标签，例如时间、距离、群体匹配、餐厅稳定性、预算、体力负担。
- UI 先展示 3 到 4 条关键推荐依据，不需要暴露复杂公式。

影响范围：

- `agent-core.js`
- `app.js`
- `styles.css`
- `tests.js`

验收：

- 同一输入下不同方案分数有合理差异。
- 推荐理由能解释为什么第一方案更优。
- 现有测试通过。

## 5. P1：增强比赛演示效果

### P1-1 强化首屏定位

目标：评委第一眼知道项目是“从一句话到可执行安排”，不是普通攻略推荐。

建议做法：

- 标题下增加副标题。
- 将 `Mock Tools` 改成更具体的状态文案：`Mock 工具：天气 / 餐厅 / 路线 / 预约`。
- 增加一行演示提示：`信息不足 -> 约束冲突 -> 亲子局 -> 确认执行`。

影响范围：

- `index.html`
- `styles.css`

验收：

- 不运行逻辑也能从首屏看懂价值。
- 移动端不出现明显拥挤或重叠。

### P1-2 增加 Agent 阶段条

目标：让工具调用和规划过程更像 Agent 工作流。

建议阶段：

- 理解需求
- 补充关键信息
- 调用工具
- 生成方案
- 用户确认
- 模拟执行

影响范围：

- `index.html`
- `app.js`
- `styles.css`

验收：

- 不改变核心规划逻辑。
- 状态随规划、追问、选择、执行而变化。

### P1-3 强化异常演示

目标：比赛现场能清楚展示“失败后重规划”。

建议做法：

- 保留餐厅无座自动替换。
- 在方案卡里把替换原因显示得更靠前。
- 增加一个“约束冲突”示例说明。
- 执行队列里区分成功、跳过、需人工确认。

影响范围：

- `agent-core.js`
- `app.js`
- `styles.css`
- `tests.js`

验收：

- 朋友局能展示餐厅无座替换。
- 约束冲突案例能展示近距离替代。
- 执行结果不得把跳过动作描述成成功。

### P1-4 扩充 Mock 数据

目标：提升真实感和方案多样性。

建议增加：

- 更多活动类型：展览、桌游、亲子剧场、室内运动、手作、咖啡、轻户外。
- 更多餐厅特征：儿童友好、低脂、长辈友好、适合聊天、可订状态、排队时长。
- 更多失败状态：无座、超时、距离过远、户外天气不佳。

影响范围：

- `agent-core.js`
- `tests.js`

验收：

- 主要示例仍能生成 2 到 3 个方案。
- 排序和推荐理由更有差异。

## 6. P2：后续增强，暂不作为第一轮重点

### P2-1 轻量多轮修改

示例：

```text
换近一点
不要西餐
预算再低一点
孩子不想走路
```

价值高，但会引入状态管理复杂度，建议在 V2 稳定后再做。

### P2-2 拆分 `agent-core.js`

当前单文件仍可维护。若继续扩展，可拆为：

- `mock-data.js`
- `parser.js`
- `planner.js`
- `tools.js`
- `executor.js`

这属于结构性改造，不建议在第一轮比赛打磨中优先做。

### P2-3 真实 API 适配层

可保留现有工具函数接口，把 Mock 实现替换为真实服务适配器。但真实 API 会带来账号、稳定性、权限和风控风险，不适合当前首轮 Demo。

## 7. 推荐执行顺序

1. 新增或调整文案：首屏定位、演示提示、Mock 工具说明。
2. 增加推荐理由摘要和维度化解释。
3. 补测试，尤其是同事局、独自探索、天气冲突、执行失败。
4. 强化异常展示：无座替换、时间距离冲突、人工确认。
5. 扩充 Mock 数据。
6. 如时间充足，再考虑轻量多轮修改。

## 8. 第一轮验收标准

- `npm test` 通过。
- Web Demo 能跑通亲子家庭主线。
- Web Demo 能展示信息不足追问。
- Web Demo 能展示约束冲突替代。
- Web Demo 能展示餐厅无座后的备选替换。
- 执行队列清楚区分高影响动作和低影响动作。
- 页面首屏能说清楚“从一句话到可执行安排”。
