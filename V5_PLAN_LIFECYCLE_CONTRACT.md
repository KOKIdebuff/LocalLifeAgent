# V5 方案保存、重排、恢复、协作与分支契约

## 状态

本文冻结首页工作台之后的方案生命周期决策。它是产品、UI Contract 和后续数据实现的共同事实源，不代表相关页面、API 或 SQLite 表已经实现。

机器可读契约位于 `ui-contract.schema.json`：

- `$defs.SavedPlanLifecycleContract`
- `$defs.SavedPlanSnapshot`
- `$defs.SelectedPlanSnapshotPayload`
- `$defs.CandidatePlanSummary`
- `$defs.PlanDetailReplanPolicy`
- `$defs.SavedPlanReopenPolicy`
- `$defs.CollaboratorSuggestionPolicy`
- `$defs.PlanBranchRecord`
- `x-savedPlanLifecycleContract`

## 1. 保存方案

保存操作只完整保存当前选中的方案，不保存整个候选集合。

完整快照必须包含：

- 当前选中方案的 `planRef`、`lineageId`、`sessionId` 和 `version`。
- 当前方案关联的 cards、entities、timeline 和可执行 actions。
- 当前假设、风险摘要、预算、时长和保存时间。
- 已采用的局部重排结果。
- 保存时的执行摘要和区块锁定状态。

其他候选方案只保存轻量摘要：

```text
planRef
name
score
recommended
rank
```

明确禁止在其他候选摘要中保存完整活动、餐厅、交通、时间线、执行队列或 cards/entities 数据。

这样做的原因：

- 保存语义明确，用户保存的是已经选中的安排。
- 避免候选集合重复保存大量结构化数据。
- 恢复时不会误把未选择候选当成稳定版本。
- 其他候选仍可用于展示“当时还比较过什么”和重新生成入口。

## 2. 行程详情继续重排

级别：重量级。

行程详情允许对活动、餐厅和时间块执行局部重排，但未修改区域必须保持引用和内容稳定。

每个可替换区块提供“换一换”，操作前必须说明影响范围。重排后必须：

- 生成新 `version`。
- 使用完整 `entityRef` 和 `targetRef`。
- 保存重排前稳定快照。
- 只更新受影响实体和时间线。
- 突出显示发生变化的内容。
- 明确标记“未保存”状态。
- 提供“撤销本次调整”。
- 校验预算、营业时间、距离、预约状态和时间冲突。

关键流程：

```text
选择区块
-> 获取稳定替代候选
-> 用户确认替代项
-> 计算时间和关联影响
-> 风险校验
-> 展示差异
-> 用户保存或撤销
```

风险规则：

- 不得静默修改无关内容。
- 重排失败必须保留原方案。
- 版本冲突时禁止覆盖。
- 替代项不可用时回退原区块。
- 高影响执行动作需要重新确认。
- 同一输入必须能确定受影响范围。
- 页面刷新后仍能恢复当前稳定版本。

分阶段落地：

1. 活动和餐厅单块替换。
2. 自动重算相邻时间线。
3. 多块级联重排。
4. 分支比较和版本合并。

第 4 阶段不属于首版。

## 3. 已保存方案重新打开

恢复策略由执行状态决定，不能把整份方案简单设为只读，也不能每次打开都覆盖刷新。

### 全部模拟成功

活动预约、餐厅订座、套餐下单等全部模拟成功时：

- 成功区块锁定为只读执行快照。
- 展示执行时的价格、时间和状态。
- 不自动重排。

### 完全没有执行

仅保存、没有模拟预约或下单时，重新打开可刷新：

- 名额。
- 餐厅可订状态。
- 套餐价格和库存。
- 营业时间。
- 时间线是否仍可执行。

发现变化后只能提示用户选择调整，不能直接覆盖原方案。

### 部分模拟成功

部分成功是最重要的边界：

- `success`：只读，展示执行快照。
- `pending`：刷新最新状态。
- `failed_recoverable`：刷新并提供替代方案。
- `cancelled`：允许重新规划。
- `skipped`：保留原信息，允许手动刷新。

局部重排不得修改已成功执行的区块，只能调整未执行部分。当前所有外部商家状态仍是 Mock 刷新，“模拟成功”不代表真实预约或下单成功。

## 4. 协作者修改建议

协作者不能直接修改共享快照，也不能覆盖当前 Main。

协作者可以：

- 查看固定分享快照。
- 对活动、餐厅或时间块提交替换建议。
- 提交整份方案反馈。
- 查看建议是否被采纳。

只有方案发起人可以：

- 采纳或拒绝建议。
- 基于建议生成 Derived Proposal。
- 将 Derived 显式采纳为新 Main。
- 发起执行。

建议记录必须包含 `suggestionId`、`shareId`、目标引用、来源快照、`baseVersion`、创建者、创建时间、幂等键、建议类型和状态。所有建议可追踪来源，重复提交需要幂等保护。分享 token 必须有权限和有效期；链接失效后进入只读，禁止继续提交。

权限边界不可简化：

- 协作者不能覆盖原始快照。
- 协作者不能触发执行、支付或模拟执行推进。
- 建议不能直接写入 Main。
- 旧版本反馈不能静默写入新版本。

## 5. Plan Branch

首版采用：

- 一个当前 `Main`。
- 多个候选 `Derived`，同时最多 3 个 active proposals。
- 显式采纳、拒绝和回滚。
- 不做任意分支合并。

用户主要看到“当前方案”和“待采纳调整”，而不是复杂 Git 式版本树。

关键流程：

```text
Main 方案
-> 局部调整或协作者建议
-> 创建 Derived
-> 展示与 Main 的差异
-> 发起人采纳或拒绝
-> 采纳后原子生成新 Main
-> 保留旧 Main 供一步回滚
```

每个 branch 必须保存：

```text
branchId
parentVersion
baseVersion
lineageId
branchType
createdBy
status
planSnapshotId
```

质量与风险规则：

- Derived 必须能追溯来源版本。
- 采纳过程必须原子化。
- Main 不能被直接覆盖。
- 旧 Main 必须可恢复。
- 禁止基于过期 Main 静默合并。
- 必须检测 base version 冲突。
- 禁止循环引用。
- 执行中的 Main 不允许被直接替换。
- 采纳和拒绝必须显式。

分阶段落地：

1. 保留 `lineageId`、`version`、父版本和 base version。
2. 实现单步撤销与上一个 Main 恢复。
3. 实现一个 Main 加一个 Derived。
4. 支持多个修改建议和最多 3 个 Derived。
5. 最后再评估复杂版本树和局部合并。

## 不采用的路径

### 保存所有候选完整数据

优点是离线回看信息最完整；缺点是存储重复、恢复语义模糊，并可能把未选方案误当成稳定快照。因此拒绝。

### 打开已保存方案时全部只读

实现简单，但未执行区块无法获得最新 Mock 状态，也无法继续完成方案。因此拒绝。

### 协作者直接修改 Main

交互看似快捷，但会破坏权限隔离、来源追踪和版本稳定性。因此拒绝。

### 完整 Git 式分支树

能力最强，但用户理解成本、冲突处理和存储复杂度过高。首版不采用。
