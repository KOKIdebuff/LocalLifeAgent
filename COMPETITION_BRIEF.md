# 比赛版项目定义：本地生活短时活动执行 Agent

## 1. 一句话定位

本项目不是普通活动推荐器，而是一个面向周末闲时的本地生活行动型 Agent：用户只说一句自然语言目标，Agent 理解同行关系、时间、偏好和限制，生成可执行的完整活动方案，并在用户确认后模拟完成预约、下单、通知和提醒等关键动作。

## 2. 参赛叙事

赛题强调“本地探索”和“帮用户把事情做完”。因此项目的核心卖点应放在从模糊目标到可执行动作的闭环，而不是只展示地点搜索或排行榜推荐。

### 2.1 赛题边界说明

本赛题不要求接入真实商家、真实地图、真实餐厅库存、真实支付/预约系统，也不要求真的爬取实时数据。因此本项目把这些外部能力统一收敛为可控的 Mock Tools，用于演示 Agent 如何理解约束、组合方案、识别风险、请求确认并模拟执行。

评审重点应放在“规划与执行闭环是否清楚可信”，而不是实时数据覆盖率、真实平台接口或真实交易能力。

评委需要看到四件事：

1. Agent 能理解人的真实约束，例如同行关系、孩子、长辈、减脂、距离、预算和体力负担。
2. Agent 会调用工具验证方案，例如天气、活动、餐厅、路线和可预约状态。
3. Agent Loop Trace 清楚展示拆解、查证、合成、校验、重排，而不是只输出推荐文案。
4. Agent 能把方案变成动作队列，并在高影响动作前保留用户确认。

## 3. 目标用户

- 周末临时想出门，但不想自己查地点、比路线、看餐厅和排时间的人。
- 有同行人约束的用户，例如朋友局、亲子家庭、情侣、带父母出门、同事下班局。
- 对安排可靠性有要求的人，例如不想太赶、不想太远、不想排队、希望餐厅能预约。

比赛 Demo 建议主打“亲子家庭”和“朋友小聚”两类用户。亲子家庭约束更丰富，能体现规划能力；朋友小聚更符合赛题截图中的即时约局场景。

## 4. 核心场景

### 主场景：亲子家庭半日安排

用户输入：

```text
今天下午想和老婆孩子出去玩 4 小时，孩子 5 岁，别太远，老婆最近在减肥。
```

Agent 需要完成：

- 识别时间是今天下午，活动时长约 4 小时。
- 识别同行关系是亲子家庭，默认人数为 3。
- 识别关键约束：别太远、孩子友好、节奏轻松、健康餐饮。
- 调用 Mock Tools 查询天气、活动、餐厅、路线和可预约状态。
- 生成 2 到 3 个候选方案，并推荐最稳方案。
- 给出时间表、活动、餐厅、预算、风险和推荐理由。
- 用户确认后模拟预约活动、预约餐厅、发送消息、设置提醒。

### 备用场景：朋友小聚即时安排

用户输入：

```text
下午和 4 个朋友出去玩，2 男 2 女，不想太累，晚上一起吃饭。
```

展示重点：

- 多人场景下的活动和餐厅匹配。
- 首选餐厅无座后的自动替换。
- 最终生成朋友群消息和执行队列。

### 判断力场景：信息不足

用户输入：

```text
今天下午有空，帮我安排一下。
```

展示重点：

- Agent 不盲猜亲子、朋友或情侣场景。
- Agent 先追问一个最关键问题，再继续规划。

### 鲁棒性场景：约束冲突

用户输入：

```text
下午 2 点出发，只能玩 1 小时，还想去很远的地方吃饭。
```

展示重点：

- Agent 能识别时间短和距离远的冲突。
- Agent 生成近距离替代方案，而不是强行给出不可执行安排。

## 5. 创新点

### 5.1 从推荐到执行闭环

普通推荐系统回答“去哪玩”。本项目回答“怎么安排，并帮你推进到可执行状态”。输出包括时间线、路线、活动、餐厅、风险、备选和执行动作。

### 5.2 自然语言目标驱动

用户不需要填写复杂表单，只说一句话。系统从自然语言中抽取时间、人数、同行关系、偏好、距离限制、餐饮诉求和冲突条件。

### 5.3 面向同行关系的策略规划

同样是“下午出去玩”，朋友局、亲子局、情侣局、长辈局的判断标准完全不同。当前实现已经按 group profile 区分不同规划策略。

### 5.4 工具调用透明可解释

页面展示 Mock API 调用链路，说明方案不是硬编码文案，而是基于天气、活动、餐厅、路线、预约状态组合出来。

### 5.5 高影响动作确认机制

预约、下单、发消息属于高影响动作，必须用户确认后才模拟执行。这让项目既体现 Agent 自动化能力，也保留真实产品需要的安全边界。

## 6. MVP 范围

第一版应聚焦在稳定展示完整闭环：

- 一句话自然语言输入。
- 关键信息不足时追问。
- 多场景识别：朋友、亲子、长辈、情侣。
- Mock Tools：天气、活动、餐厅、路线、可预约状态。
- 候选方案生成和推荐。
- 风险说明和备选替换。
- 用户确认后的执行队列。
- 无依赖本地 Web Demo 和回归测试。

## 7. 第一版不做

- 不接入真实美团、大众点评、携程、真实商家、真实地图、支付或即时通讯平台。
- 不接入真实餐厅库存、真实预约系统或真实商家经营数据。
- 不做真实支付、真实出票、真实下单或真实预约。
- 不做跨城市旅行规划，聚焦本地短时活动。
- 不做复杂登录、账号、历史画像和长期日程管理。
- 不做多人协同投票。
- 不做实时数据爬取或大规模真实数据抓取。
- 不做无确认的自动预约、自动支付或自动发送消息。

## 8. Demo 讲解脚本

1. 先说明项目定位：从一句话目标到可执行本地生活安排。
2. 输入亲子家庭主场景，点击生成方案。
3. 展示理解结果：时间、同行关系、人数、偏好和假设。
4. 展示 Agent Loop Trace：Planner 拆解需求，Researchers 查天气/活动/餐厅/路线/订座，Merger 合成服务包，Verifier 校验可执行性，Revise 准备重排。
5. 展示原始 Mock Tool 调用，说明不是硬编码推荐文案。
6. 展示推荐方案：为什么这个方案更近、更适合孩子、更符合减脂餐饮、更可预约。
7. 展示备选方案：说明不是单点推荐。
8. 点击确认执行，展示预约、消息、提醒等动作结果。
9. 切换“信息不足”案例，展示 Agent 会停在 Planner 追问，不假装进入 Researchers。
10. 切换“约束冲突”或“餐厅满座”案例，展示 Verifier / Revise 如何给出替代方案。

## 9. 当前状态

当前更准确的说法是：项目的比赛主链路仍然是 V3 风格的本地 Mock 执行型 Agent，但仓库里已经额外落地了一部分 V4 alpha 增强能力。

当前已稳定可讲的主链路包括：

- 静态 Web Demo。
- Mock Tools。
- Agent Loop Trace。
- 服务包生成。
- 闲时错峰。
- 动态重排。
- 执行队列。
- 回归测试。

主线已经从“推荐去哪玩”推进到“拆解、查证、合成、校验、重排并生成可确认的复合周末局服务包”，覆盖活动、餐厅、路线、排队、订座、团购、加购、通知和提醒。

同时，仓库里已经存在但不应在比赛叙事中夸大的增强能力包括：

- 可选 FastAPI 后端。
- `/api/intent` LLM 意图识别入口。
- `/api/feedback` 和 `/api/memory-candidates/{id}/decision` 反馈复盘入口。
- SQLite 记忆表与 JSONL 审计日志。
- 前端 `Reflect` 阶段和反馈面板。

因此，对外最稳妥的表达应是：

- 比赛主链路仍以本地 Mock 为主。
- 后端与记忆闭环属于已落地但仍在收敛中的 alpha 能力。

V3 冲刺阶段不建议推倒重来，也不建议临时改成重型多 Agent 框架。比赛前应重点检查演示路径、强化讲解脚本和视觉层级，并持续明确当前所有外部平台能力均为本地 Mock。

## 10. 下一阶段 V4 路线图

V4 仍然不是当前已经完整完成的 Runtime。更准确地说，当前只是已经提前落地了部分 alpha 组件，下一阶段要做的是把这些组件收敛成更稳定的轻量 Agent Runtime。

目标仍然是从“稳定 Mock 演示”升级为“轻量 Agent Runtime + 真实 LLM 意图识别 + 可复盘记忆”，真实提升自然语言泛化能力，同时保持高影响动作可确认、可解释、可回滚。

V4 不使用 Claude Code 等任何非官方泄露源码，只参考 Claude Code、Codex、Cursor 等公开产品形态中的通用 Agent 思想，例如任务循环、工具调用、权限确认、记忆、Trace、复盘和子任务拆解。实现上仍保持自有轻量架构，不优先引入完整重型 Agent 框架。

建议架构：

```text
User Input
  -> IntentExtractor        # LLM 结构化理解自然语言
  -> IntentValidator        # Schema 校验、置信度、缺字段和冲突检测
  -> StateMachine           # understand / ask / research / plan / verify / execute / reflect
  -> ToolRegistry           # Mock Tools 与未来真实 API 的统一接口
  -> Planner + Verifier     # 复用当前服务包生成、评分和风险校验
  -> Reflector              # 总结用户纠错、执行失败和误判原因
  -> MemoryStore            # 候选确认后沉淀结构化记忆，下次检索参考
```

V4 的“自进化”定义应保持克制：系统会自动总结用户纠正、方案失败和工具异常经验，先生成候选记忆，由用户采用/忽略/更正后才进入长期记忆；后续类似输入会检索参考，用于降低重复错误概率。不表述为自动改代码，也不承诺永远不会犯同样错误。

记忆存储边界：

- SQLite 是事实库，保存反馈、候选记忆、长期记忆和记忆引用记录。
- JSON 是前后端交换格式。
- JSONL 是审计日志，用于记录反馈、候选决策和检索事件。
- Vector DB 只作为后续语义检索增强，不作为 V4 第一版依赖。
- 默认只长期保存 L0/L1 抽象偏好；L2/L3 高敏感信息默认不进入长期记忆。

V4 优先任务：

1. 把当前已存在的后端入口继续收敛，而不是另起一套并行实现。
2. 定义 `intent.schema.json`，约束场景、人数、时间、预算、偏好、缺失字段和置信度。
3. 保留当前规则解析作为 LLM 失败兜底。
4. 把现有 `/api/feedback` 和 `/api/memory-candidates/{id}/decision` 的契约、状态与测试补齐。
5. 继续完善 `Reflect` 阶段，让“复盘经验如何进入下次规划”表达得更清楚。
